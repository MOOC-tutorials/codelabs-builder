En este proyecto vamos a practicar el conjunto de comandos básicos de git que se han aprendido hasta el momento. La tarea consiste en realizar, junto con un equipo de trabajo automatizado, la corrección de una página web estática que presenta múltiples errores. Es prioridad de su equipo lograr corregirlos.

El código se encuentra disponible en Github y a continuación se explica brevemente la configuración del ambiente de desarrollo necesaria para comenzar a trabajar. Posteriormente se explica la manera en que se le informará de los errores a corregir y las soluciones a implementar.

Antes de comenzar

  • Asegúrese de tener una cuenta en Github y tener la sesión iniciada.
  • Asegurese de tener Git instalado.
  • Se recomienda usar algún editor de archivos con manejo de sintaxis. Una opción recomendada es usar Visual Studio Code.

Primeros pasos

  • Ir a https://github.com/MOOC-tutorials/git_web_practice
  • Dar clic sobre el botón de
  • Confirmar que ahora tenga dentro de su cuenta un nuevo repositorio con el sitio web estático base con el nombre git_web_practice

Dando permisos al equipo de trabajo automático

  • Instalar la Github App disponible en https://github.com/apps/static-web-project
  • Clic en
  • Seleccionar su cuenta (donde quedó creado el repositorio git_web_practice)
  • Seleccionar, hacer clic sobrey seleccionar el repositorio git_web_practice. Clic en

Clonando el repositorio

  • En su repositorio git_web_practice en Github, hacer clic en
  • Copiar la URL dada. Puede usar el botón (este botón copiará en su portapapeles la URL que se necesita).
  • En su terminal/consola ubicarse en el directorio en que desea que se clone el repositorio usando el comando cd.
  • En su terminal/consola escribir git clone <URL del repositorio>.
  • Cambiar el directorio activo (cd) a la carpeta raíz del repositorio. Ejecutando cd git_web_practice debería bastar.

En este momento debería de contar con una copia local del repositorio con el código de la página web estática. Partiendo de esto ahora podrá hacer los cambios necesarios para corregir los errores. La idea general es arreglar este sitio teniendo en cuenta las instrucciones que irán apareciendo en la sección de issues en el repositorio en Github. Por cada arreglo especificado es necesario hacer un commit. Además, luego de cada commit, debe subir sus cambios locales mediante un push al repositorio remoto.

Instrucciones sobre los errores a corregir

Podrá ir observando los errores a corregir en su repositorio en Github (aparecerán dentro de sus issues asignados con el tag bug). Para empezar a trabajar, diríjase a la sección de issues de su repositorio. Allí encontrará el primer error a corregir.

A partir de este momento cualquier información o acción a realizar sobre el repositorio o su trabajo le será reportada mediante un issue en la sección correspondiente en Github del repositorio.

Para el correcto desarrollo de la actividad, tenga en cuenta cada uno de los siguientes puntos:

  • Por cada error debe realizar un commit cuyo comentario siga la convención: <FIX ID>: <Comentario>. Por ejemplo para el ISSUE 1 un comentario adecuado para el commit de resolución sería: FIX1: Corrección de páginas 1 y 5

Cumplir con esta convención es de suma importancia para el desarrollo de la actividad y la recepción de una correcta retroalimentación.

  • Serán asignados tres (3) issues mientras va trabajando. Al finalizar un issue, se le asignará el siguiente y podrá encontrarlo en el repositorio en Github (puede que sea necesario refrescar la página luego de finalizada una tarea previa). Si la corrección no es la esperada, se le informará y deberá realizar la revisión y cambios que correspondan y volver a realizar el commit de los cambios.
  • Puede que durante la corrección surjan conflictos. En dicho caso, debe de corregirlos y realizar el commit con el comentario correspondiente al fix que estaba implementando en el momento que el conflicto surgió con la siguiente convención: FIX<ID>: <Comentario por defecto del merge>
  • En caso de querer volver a comenzar el ejercicio asegúrese de eliminar el fork creado y no tener ningún repositorio con el mismo nombre que el repositorio base usado dentro de la actividad.
